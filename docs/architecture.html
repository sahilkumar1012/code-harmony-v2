Data Flow

Static data lives in src/data/ as plain JavaScript arrays/objects. Each component imports directly from these files — no API calls, no Redux. Firebase is used only for authentication and optional Firestore persistence. 

src/data/mentors.js ──────→ MentorShowcase, Mentors page
src/data/reviews.js ─────→ TestimonialSlider
src/data/faqs.js ────────→ FAQAccordion
src/data/videos.js ──────→ YouTubeSection
src/data/services.js ────→ ServicesGrid

Firebase Auth ───────────→ AuthProvider (context)
                              ├── Navbar (avatar / login btn)
                              └── Login page (Google popup)
💡 Tip: To add a new mentor, edit src/data/mentors.js and the grid updates automatically — no API or database change required.

Deployment Pipeline

Code Harmony v2 deploys to Firebase Hosting in three simple steps:

┌──────────────┐     ┌──────────────┐     ┌──────────────────────┐
│  npm run     │     │   dist/      │     │  firebase deploy     │
│  build       │ ──→ │   (static)   │ ──→ │  (Firebase Hosting)  │
└──────────────┘     └──────────────┘     └──────────────────────┘
  1. Run npm run build — Vite outputs optimised static files to dist/.
  2. Ensure firebase.json has "public": "dist" and SPA rewrite rules.
  3. Run firebase deploy — uploads dist/ to Firebase Hosting CDN.
✅ Result: Live at codeharmonyapp.web.app with global CDN, free SSL, and instant cache invalidation.

Key Design Decisions

DecisionRationale
Static data files over Firestore for content Content changes infrequently; avoids read costs and cold-start latency
Tailwind CSS over CSS Modules Faster iteration, smaller bundle with purge, consistent design tokens
Vite over CRA / Webpack Near-instant HMR, faster builds, native ESM support
Context API over Redux Only two global states (auth + theme) — Redux would be over-engineering
Glassmorphism Navbar Modern visual appeal; backdrop-filter is well-supported in target browsers
